# Temporal Architecture Benchmark Checklist

This document tracks evidence for the engineering story behind the Temporal parser architecture: deterministic validation plus an intermediate Plan-IR generated by a tuned small language model.

## Narrative Hypothesis

- One-shot large-model parsing should be accurate but too slow and expensive for a desktop overlay target.
- Deterministic parsing should be fast and reliable for explicit syntax, but it should not absorb fuzzy language interpretation through ad hoc heuristics.
- Untuned small models should be fast enough locally but unreliable at producing valid, faithful structured outputs.
- A fine-tuned small language model targeting a narrow intermediate representation should provide the best tradeoff: bounded latency, measurable accuracy, deterministic execution, and understandable failures.

## Product Targets

- [ ] End-to-end parse latency stays under 5 seconds for interactive desktop use.
- [ ] Common explicit cases resolve deterministically without model calls.
- [ ] Fuzzy cases either produce valid Plan-IR or fail closed with a useful fallback/no-plan response.
- [ ] `time_range` outputs preserve ordered start/end semantics and Discord formatting.
- [ ] Timezone and DST behavior remains deterministic and auditable.

## Baselines To Benchmark

- [ ] Deterministic-only parser.
- [ ] Existing large-model one-shot parser.
- [ ] Existing mini/nano model one-shot parser.
- [ ] Plan-IR through a large model.
- [ ] Local 0.8B model without fine-tuning, minimal prompt.
- [ ] Local 0.8B model without fine-tuning, detailed prompt.
- [ ] Local fine-tuned 0.8B Plan-IR SLM.
- [ ] Hosted fine-tuned Plan-IR SLM, warm endpoint.
- [ ] Hosted fine-tuned Plan-IR SLM, cold endpoint.

## Metrics To Capture

- [ ] Required eval pass rate.
- [ ] Diagnostic eval pass rate.
- [ ] Invalid JSON / invalid schema rate.
- [ ] Valid schema but wrong answer rate.
- [ ] No-plan / clarification rate.
- [ ] End-to-end latency p50, p90, p95, and max.
- [ ] Model-only latency p50, p90, p95, and max.
- [ ] Deterministic execution latency.
- [ ] Token usage and estimated cost for hosted model paths.
- [ ] Local GPU memory footprint for local SLM paths.
- [ ] Cold-start latency for hosted/container paths.

## Dataset Slices

- [ ] Explicit dates and times.
- [ ] Relative dates and times.
- [ ] Bare clocks.
- [ ] Bare clock ranges, such as `5pm-7pm`.
- [ ] Existing Discord timestamp instants.
- [ ] Existing Discord timestamp ranges.
- [ ] Timezone abbreviations and IANA zones.
- [ ] Ambiguous timezone abbreviations.
- [ ] DST gaps and folds.
- [ ] Intentionally underspecified prompts that should clarify or no-plan.
- [ ] Negative examples that should not produce timestamps.

## Failure Classes

- [ ] Wrong date arithmetic.
- [ ] Wrong timezone resolution.
- [ ] DST invalid instant accepted.
- [ ] `time_range` end before start.
- [ ] Hallucinated fields in Plan-IR.
- [ ] Missing required Plan-IR fields.
- [ ] Natural-language answer instead of JSON/Plan-IR.
- [ ] Deterministic fallback masks model failure incorrectly.
- [ ] Slow response exceeds product latency target.
